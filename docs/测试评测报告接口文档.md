# 测试评测报告接口文档（前端对接版）

## 1. 文档目的

本文档用于指导前端同学或前端开发 Agent 对接新的“测试评测结果报告”页面。

本文档重点说明：

1. 报告页面每个区域调用哪个接口；
2. 页面上的每个指标对应响应中的哪个字段；
3. 接口的详细入参和出参；
4. 哪些页面内容已实现，哪些内容按需求不实现；
5. 前端需要对原 HTML 示例做哪些字段口径调整。

对应代码分支：

```text
codex/report-page-apis
```

当前服务器由网关增加 `/api` 前缀，因此本文示例使用：

```text
http://183.66.251.10:60099/api
```

应用代码内部实际路由不包含 `/api`。

## 2. 实现范围确认

### 2.1 已实现的报告功能

| 报告页面区域 | 是否实现 | 后端数据来源/接口 |
| --- | --- | --- |
| 报告标题、测试对象 | 已实现 | 报告汇总接口 `test_object` |
| 评价概况 | 已实现 | 报告汇总接口 `overview` |
| 综合评分、客观评分、专家平均分 | 已实现 | 报告汇总接口 `overview` |
| 测试是否通过、判定依据 | 已实现 | 报告汇总接口 `overview` |
| 用例执行汇总 | 已实现 | 报告汇总接口 `execution_summary` |
| 测试用例明细 | 已实现 | 测试用例分页接口 |
| 缺陷等级统计 | 已实现 | 报告汇总接口 `defect_statistics.severity_distribution` |
| 缺陷场景/频率统计 | 已实现 | 报告汇总接口 `defect_statistics` |
| 缺陷明细清单 | 已实现 | 缺陷分页接口 |
| 整体质量分析 | 已实现 | 大模型分析接口 `quality_analysis` |
| 综合结论 | 已实现 | 大模型分析接口 `conclusion` |
| 遗留风险分析 | 已实现 | 大模型分析接口 `residual_risks` |
| 后续优化落地建议 | 已实现 | 大模型分析接口 `optimization_suggestions` |
| 专家列表 | 已实现 | 专家列表接口 |
| 邀请专家评分 | 已实现 | 专家邀请接口，复用原邮件发送逻辑 |
| 下载 Word 报告 | 由前端实现 | 原 HTML 已使用 `html-docx-js` 和 `FileSaver`，不需要后端接口 |

### 2.2 明确不实现的功能

以下区域按需求不做，前端应从新报告页删除或隐藏：

```text
测试覆盖度统计
```

后端没有提供需求总数、P0/P1/P2 覆盖率、非功能需求覆盖率等接口字段。

### 2.3 相对原 HTML 的两项指标调整

#### 缺陷等级

原 HTML 示例使用 `S/A/B/C`，当前项目真实数据使用：

```text
A / B / C / D
```

前端应按接口返回的动态键展示，不要写死为 `S/A/B/C`。

#### 缺陷状态

原 HTML 中包含：

- 缺陷状态分布图；
- 缺陷明细中的“已修复、已确认、已激活、重新打开、已关闭”。

当前上游数据没有缺陷状态字段，因此不能可靠生成这些数据。本次接口调整为：

- 缺陷状态分布图 → 优先使用缺陷发生频率分布；上游提供规范场景枚举时也可使用缺陷场景分布；
- 缺陷明细“缺陷状态” → “发生频率”或“缺陷场景”。

## 3. 原有接口说明

原接口未修改：

```http
GET /api/tpa/get_testset_by_data_id?data_id={data_id}
```

新报告页建议使用本文档中的新接口。新接口没有修改原接口的地址、入参和出参。

## 4. 新增接口总览

| 序号 | 方法 | 接口 | 用途 |
| --- | --- | --- | --- |
| 1 | GET | `/api/tpa/report/summary?data_id={data_id}` | 报告标题、评价概况、执行汇总、缺陷统计、分析状态 |
| 2 | GET | `/api/tpa/report/testcases?data_id={data_id}` | 分页查询测试用例明细 |
| 3 | GET | `/api/tpa/report/defects?data_id={data_id}` | 分页查询缺陷明细 |
| 4 | GET | `/api/tpa/report/experts` | 查询可邀请的评分专家 |
| 5 | POST | `/api/tpa/report/invitations` | 邀请专家评分 |
| 6 | GET | `/api/tpa/report/analysis?data_id={data_id}` | 查询大模型分析状态和结果 |
| 7 | POST | `/api/tpa/report/analysis/generate` | 触发大模型生成分析 |

所有需要 `data_id` 的 GET 接口均使用 Query 参数，不使用 URL 路径参数。

## 5. 页面区域与接口字段映射

这是前端对接时最重要的部分。

### 5.1 页面头部

调用：

```http
GET /api/tpa/report/summary?data_id={data_id}
```

| 页面内容 | 接口字段 |
| --- | --- |
| 报告标题“测试评测结果报告” | 前端固定文案 |
| 测试对象 | `test_object` |
| “专家评测”等副标题 | 前端按产品需求组合；可根据 `overview.expert_average_score` 是否为空判断是否已有专家评分 |
| 下载 Word | 前端原有 `downloadReport()`，不调用后端 |

### 5.2 评价概况

调用报告汇总接口，使用 `overview`：

| 页面内容 | 接口字段 | 说明 |
| --- | --- | --- |
| 综合评分 | `overview.comprehensive_score` | 可能为 `null` |
| 客观评分 | `overview.objective_score` | 可能为 `null` |
| 专家平均分 | `overview.expert_average_score` | 没有专家评分时为 `null` |
| 测试通过/未通过 | `overview.status_text` | `测试通过` 或 `测试未通过` |
| 是否通过布尔值 | `overview.passed` | 建议用于控制样式 |
| 测试用例总数 | `overview.testcase_total` | 整数 |
| 整体通过率 | `overview.overall_pass_rate` | 数值，前端补 `%` |
| 缺陷总数 | `overview.defect_total` | 按 `bug_id` 去重后的数量 |
| 判定依据 | `overview.judgement_reasons` | 字符串数组 |
| 是否存在 A/B 类阻断缺陷 | `overview.has_blocking_defect` | 布尔值 |
| 通过分数线 | `overview.pass_threshold` | 当前为 90 |

### 5.3 用例执行汇总

调用报告汇总接口，使用：

```text
execution_summary
```

每一项对应表格一行：

| 页面列 | 接口字段 |
| --- | --- |
| 用例优先级 | `priority` |
| 用例数量 | `total` |
| 测试通过用例数 | `passed` |
| 测试失败用例数 | `failed` |
| 异常用例数 | `abnormal` |
| 无判定用例数 | `undetermined` |
| 未执行用例数 | `not_executed` |
| 测试通过率 | `pass_rate`，前端补 `%` |

返回行可能包括：

```text
bvt优先级
高优先级
中优先级
低优先级
全部
```

前端可将 `priority=全部` 的记录作为汇总行。

### 5.4 测试用例明细

调用：

```http
GET /api/tpa/report/testcases?data_id={data_id}&page=1&page_size=20
```

| 原 HTML 表格列 | 接口字段 | 前端处理 |
| --- | --- | --- |
| 用例编号 | `testcase_number` | 直接展示 |
| 用例名称 | `testcase_name` | 直接展示 |
| 测试类型 | `testcase_type` | 直接展示 |
| 优先级 | `priority` | 已转换为中文 |
| 描述信息 | `description` | 直接展示 |
| 逻辑用例 | `logical_cases` | 数组，前端按实际子字段格式化 |
| 测试步骤 | `steps` | 按 `order` 排序，拼接 `action` |
| 预期结果 | `expected_results` | 字符串数组，可换行展示 |

接口还返回测试集、前置条件、执行结果和关联缺陷，可用于扩展展示或筛选。

### 5.5 缺陷分布统计

调用报告汇总接口，使用：

```text
defect_statistics
```

| 页面内容 | 接口字段 |
| --- | --- |
| 缺陷总数 | `defect_statistics.total` |
| 缺陷等级分布表 | `defect_statistics.severity_distribution` |
| 缺陷场景分布图 | `defect_statistics.scenario_distribution`；HTML 或过长内容统一归入“未分类” |
| 缺陷发生频率分布图 | `defect_statistics.frequency_distribution`，当前推荐使用 |

原 HTML 的“缺陷状态分布图”建议替换为“缺陷发生频率分布图”。`scenario_distribution` 只有在上游 `defect_scenario` 是规范短文本枚举时才适合作图；HTML、测试步骤或超过 64 个字符的值会统一统计为“未分类”。

分布字段是动态对象，例如：

```json
{
  "A": 2,
  "B": 3,
  "C": 1,
  "D": 0
}
```

前端应遍历对象键值，不应写死固定等级。

### 5.6 缺陷明细清单

调用：

```http
GET /api/tpa/report/defects?data_id={data_id}&page=1&page_size=20
```

| 原 HTML 表格列 | 接口字段 | 说明 |
| --- | --- | --- |
| 所属测试集 | `testsuite_names` | 数组，可能关联多个测试集 |
| 缺陷编号 | `bug_id` | 缺陷唯一 ID |
| 关联用例编号 | `associated_testcase_ids` | 数组，可能关联多个用例 |
| 缺陷描述 | `title` | 缺陷标题 |
| 严重程度 | `severity` | A/B/C/D |
| 缺陷状态 | 不提供 | 替换为 `frequency` 或 `defect_scenario` |
| 发生频率 | `frequency` | 建议作为新列 |
| 缺陷场景 | `defect_scenario` | HTML 或过长脏数据会返回“未分类” |
| 复现步骤 | `reproduction_steps` | 直接展示 |
| 所属模块 | `module` | 可作为扩展列 |

### 5.7 测试覆盖度统计

不实现。

前端应删除或隐藏整个“测试覆盖度统计”区域，不调用后端。

### 5.8 整体质量分析和综合结论

数据由一次同步大模型调用统一生成。前端调用生成接口后需要等待请求完成，成功响应会直接包含整体质量分析、综合结论、遗留风险和优化建议。

先查询：

```http
GET /api/tpa/report/analysis?data_id={data_id}
```

生成成功后：

| 页面内容 | 接口字段 |
| --- | --- |
| 整体质量分析卡片 | `quality_analysis` |
| 卡片标题 | `quality_analysis[].dimension` |
| 优/良/中/差标签 | `quality_analysis[].level` |
| 分析说明 | `quality_analysis[].summary` |
| 综合结论 | `conclusion` |

### 5.9 遗留风险分析

调用大模型分析查询接口，使用：

```text
residual_risks
```

| 页面列 | 接口字段 |
| --- | --- |
| 风险编号 | `risk_id` |
| 风险描述 | `description` |
| 风险等级 | `level` |
| 关联缺陷/用例 | `related_items` |
| 影响范围 | `impact_scope` |
| 应对措施与建议 | `mitigation` |

### 5.10 后续优化落地建议

调用大模型分析查询接口，使用：

```text
optimization_suggestions
```

| 页面内容 | 接口字段 |
| --- | --- |
| 建议标题 | `title` |
| 建议内容 | `content` |

### 5.11 专家邀请抽屉

打开抽屉时调用：

```http
GET /api/tpa/report/experts
```

| 页面内容 | 接口字段 |
| --- | --- |
| 专家姓名 | `name` |
| 专家邮箱 | `email` |
| 头像文字 | 前端可取 `name` 首字符 |

用户确认邀请时调用：

```http
POST /api/tpa/report/invitations
```

## 7. 接口二：分页查询测试用例

### 7.1 请求

```http
GET /api/tpa/report/testcases
```

### 7.2 入参

| 参数 | 位置 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `data_id` | Query | string | 是 | - | 测试评测任务唯一 ID |
| `page` | Query | integer | 否 | `1` | 页码，最小为 1 |
| `page_size` | Query | integer | 否 | `20` | 每页数量，范围 1～200 |
| `priority` | Query | string | 否 | - | 优先级，支持原始值 `1/2/3/4` 或中文值 |
| `testsuite_id` | Query | string | 否 | - | 按测试集 ID 精确筛选 |
| `keyword` | Query | string | 否 | - | 模糊匹配用例 ID、编号、名称、模块或描述 |

优先级映射：

| 参数值 | 含义 |
| --- | --- |
| `1` 或 `低优先级` | 低优先级 |
| `2` 或 `中优先级` | 中优先级 |
| `3` 或 `高优先级` | 高优先级 |
| `4` 或 `bvt优先级` | BVT 优先级 |

### 7.3 示例

```bash
curl 'http://183.66.251.10:60099/api/tpa/report/testcases?data_id=b79db457633a43eea6d59dd4d9b02270&page=1&page_size=20&priority=高优先级'
```

### 7.6 测试用例字段

| 字段 | 类型 | 说明 |

## 12. 接口七：触发大模型分析

### 12.1 请求

```http
POST /api/tpa/report/analysis/generate
Content-Type: application/json
```

### 12.2 请求体

```json
{
  "data_id": "b79db457633a43eea6d59dd4d9b02270",
  "force": false
}
```

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `data_id` | string | 是 | - | 测试评测任务唯一 ID |
| `force` | boolean | 否 | `false` | 是否强制重新生成已有成功结果 |

### 12.3 同步调用说明

该接口为同步接口。请求会一直等待大模型返回、JSON 结构校验和 MongoDB 持久化全部完成。

成功时 HTTP 状态码为 `200 OK`，直接返回完整分析结果：

```json
{
  "data_id": "b79db457633a43eea6d59dd4d9b02270",
  "status": "ready",
  "quality_analysis": [
    {
      "dimension": "核心稳定性",
      "level": "良",
      "summary": "核心流程总体稳定，但仍存在高优先级失败用例。"
    }
  ],
  "conclusion": "完成高等级缺陷修复和回归后再进行准出评审。",
  "residual_risks": [],
  "optimization_suggestions": [
    {
      "title": "加强高优先级回归",
      "content": "修复缺陷后对关联用例执行专项回归。"
    }
  ],
  "generated_at": "2026-07-08T10:35:00+00:00",
  "updated_at": "2026-07-08T10:35:00+00:00"
}
```

模型调用失败时返回 `502 Bad Gateway`，同时失败状态会写入 `report_analyses`：

```json
{
  "detail": "大模型分析生成失败: 具体错误信息"
}
```

如果另一个请求正在生成同一份报告，返回 `409 Conflict`：

```json
{
  "detail": "该报告正在生成大模型分析，请勿重复提交"
}
```

### 12.4 `force` 行为

| 当前状态 | `force=false` | `force=true` |
| --- | --- | --- |
| `not_generated` | 创建任务 | 创建任务 |
| `failed` | 重新创建任务 | 重新创建任务 |
| `ready` | 返回已有结果 | 重新创建任务 |
| `processing` | 返回 409 | 返回 409 |

大模型生成内容不包含测试覆盖度统计。

> 同步请求能否完整等待取决于部署超时配置。当前 Gunicorn 默认超时为 300 秒，Nginx 或其他网关的读取超时也必须大于模型实际耗时，否则模型仍可能在服务端执行，但客户端会提前收到超时。

## 13. 通用错误响应

### 13.1 报告不存在：HTTP 404

```json
{
  "detail": "data_id=b79db457633a43eea6d59dd4d9b02270 的测试评测报告不存在"
}
```

表示 `processed_testsets` 中不存在对应报告。

### 13.2 参数校验失败：HTTP 422

```json
{
  "status_code": 422,
  "message": "参数校验错误信息",
  "data": null
}
```

常见原因：

- 缺少 `data_id`；
- `page` 小于 1；
- `page_size` 小于 1 或大于 200；
- 专家邀请的 `emails` 为空；
- 请求体不是合法 JSON。

### 13.3 路由不存在：HTTP 404

```json
{
  "detail": "Not Found"
}
```

这表示 URL 没有匹配到后端路由，不表示 `data_id` 没数据。应检查是否使用本文档中的 `/api/tpa/report/...` 地址。

## 14. 前端推荐调用流程

### 14.1 页面首次加载

并行调用：

```http
GET /api/tpa/report/summary?data_id={data_id}
GET /api/tpa/report/testcases?data_id={data_id}&page=1&page_size=20
GET /api/tpa/report/defects?data_id={data_id}&page=1&page_size=20
```

报告汇总接口用于渲染：

- 页面头部；
- 评价概况；
- 用例执行汇总；
- 缺陷分布统计；
- 当前大模型分析状态。

### 14.2 大模型分析

读取汇总响应中的：

```text
analysis.status
```

处理方式：

1. `not_generated`：调用同步分析生成接口，并显示页面级加载状态；
2. 生成接口返回 `ready`：直接使用响应渲染整体质量分析、综合结论、遗留风险和优化建议；
3. `processing`：说明另一个请求正在生成，不要重复提交，可稍后调用查询接口；
4. `ready`：直接渲染已保存的分析结果；
5. `failed`：显示错误并允许用户重新生成。

### 14.3 专家邀请

1. 用户打开专家抽屉时调用专家列表接口；
2. 前端维护勾选邮箱列表；
3. 用户确认后调用专家邀请接口。

### 14.4 Word 下载

继续使用报告页 HTML 中已有的前端导出逻辑，无需调用后端。

### 14.5 测试覆盖度

不渲染、不调用接口。
